# TROUBLE — company·position 다중 카테고리 적용 버그

`api/analyze.js`에서 회사명(company)과 직군(position)을 같은 문장 안에서 함께 처리할 때 발견한 두 가지 버그와 해결 과정을 정리한다.

---

## 1. `original`이 짧은 구절이라 엉뚱한 위치에 적용됨

### 증상

아래 입력으로 테스트했을 때, 경력을 설명하는 앞 문장의 직무명이 잘못 바뀌고 정작 포부를 나타내는 뒷문장의 직무명은 그대로 남았다.

```
입력: "삼성전자에서 개발자로 근무했습니다. 카카오에서 개발자로 근무하고 싶습니다."
```

```json
{
  "id": "position",
  "original": "개발자로",
  "suggestion": "프로덕트 디자이너로"
}
```

### 원인

본문에 "개발자로"라는 문구가 **두 번**(경력 문장·포부 문장) 등장한다. 코드는 `text.indexOf(original)`로 제안이 적용될 위치를 찾는데, 이는 항상 **가장 먼저 나오는 위치**를 반환한다. AI는 뒷문장(포부 문장)을 의도하고 이 제안을 만들었지만, `original`이 "개발자로"라는 짧은 조각뿐이라 코드는 이를 구분하지 못하고 앞문장(경력 문장)에 잘못 적용했다.

company 제안은 이 문제가 없었는데, `original`을 코드가 직접 "문장 전체" 단위로 만들어서 넘겨주기 때문이다. 반면 position/keyword/grammar/tone은 AI가 `original`을 자유롭게 정하다 보니, 이번처럼 문장 전체가 아니라 일부 구절만 반환하는 경우가 있었다.

### 해결

company뿐 아니라 keyword/position/grammar/tone 모두 `original`을 반드시 "그 수정이 포함된 문장 전체"로 반환하도록 프롬프트 공통 규칙에 명시했다.

```
- original 은 반드시 그 수정이 포함된 문장 전체를 옮겨 적어라. 짧은 구절만 쓰면 안 된다 —
  본문에 같은 표현이 여러 번 나올 경우 어느 위치를 말하는지 구분할 수 없기 때문이다.
```

화면 카드에는 `diffSpan()`이 문장에서 실제로 바뀐 부분만 잘라서 보여주고 있으므로, `original`이 문장 전체로 길어져도 사용자에게 보이는 표시에는 영향이 없다.

---

## 2. 한 카테고리의 suggestion이 다른 카테고리의 담당 영역까지 건드려서 겹침 처리에서 탈락함

### 증상

1번 문제를 고친 뒤에도, 뒷문장에서 회사명은 정확히 바뀌었는데 직무명은 여전히 반영되지 않는 경우가 있었다.

```json
[
  {
    "id": "company",
    "original": "카카오에서 개발자로 근무하고 싶습니다.",
    "suggestion": "토스에서 개발자로 근무하고 싶습니다."
  },
  {
    "id": "position",
    "original": "카카오에서 개발자로 근무하고 싶습니다.",
    "suggestion": "토스에서 프로덕트 디자이너로 근무하고 싶습니다."
  }
]
```

최종 결과: `삼성전자에서 개발자로 근무했습니다. 토스에서 개발자로 근무하고 싶습니다.` → 회사명만 바뀌고 직무명("개발자")은 그대로 남음.

### 원인

position의 `suggestion`을 보면 직무명뿐 아니라 회사명("토스")까지 같이 자연스러운 문장으로 다시 썼다. `original`("카카오에서...")과 `suggestion`("토스에서...")을 비교해 실제로 달라진 구간을 계산(`diffSpan`)하면, 회사명 부분까지 포함해서 훨씬 넓게 잡힌다. 이 넓어진 범위가 company가 이미 차지한 범위("카카오"→"토스" 부분)와 **글자 위치가 겹치게** 되고, 겹치는 두 수정 중 우선순위(`CATEGORY_ORDER`)가 높은 company만 남기고 position은 통째로 버려지는 로직(`computeSuggestionEdits`) 때문에 직무명 수정이 사라졌다.

### 해결

프롬프트 공통 규칙에 "각 카테고리는 자기 담당 부분만 수정하고, 나머지(특히 다른 카테고리가 담당하는 회사명 등)는 원문 그대로 유지하라"를 명시했다.

```
- suggestion 을 작성할 때는 그 카테고리가 담당하는 부분만 수정하고, 나머지는 original 그대로 유지해라.
  특히 회사명은 이 프롬프트가 담당하지 않으니(별도로 처리됨) 절대 바꾸지 말고 원문 그대로 남겨라.
```

이렇게 하면 position의 `suggestion`은 "카카오에서 **프로덕트 디자이너**로 근무하고 싶습니다."처럼 회사명은 그대로 두고 직무명만 바뀐 형태로 나오고, `diffSpan()`이 잡는 범위도 직무명 부분으로 좁아져 company의 수정 범위와 겹치지 않는다.

### 검증 결과

수정 후 동일 입력으로 5회 반복 테스트한 결과, 경력 문장의 직무명은 5/5 그대로 유지되고 포부 문장의 회사명·직무명은 5/5 모두 정확히 함께 반영됨을 확인했다.

```
최종 결과: 삼성전자에서 개발자로 근무하였습니다. 토스에서 프로덕트 디자이너로 근무하고 싶습니다.
```

---

## 교훈

- 여러 카테고리가 같은 문장을 다룰 때는 **글자 위치까지 정확히 계산**해야 진짜 겹침(같은 글자를 두 번 건드리는 것)과 가짜 겹침(같은 문장이지만 다른 부분을 건드리는 것)을 구분할 수 있다.
- AI에게 "자연스러운 문장으로 고쳐라"라고만 하면, 담당 범위를 넘어서 다른 부분까지 같이 재작성하는 경향이 있다 — 카테고리별로 "이 부분만 고치고 나머지는 그대로 두라"는 제약을 명시적으로 줘야 한다.
