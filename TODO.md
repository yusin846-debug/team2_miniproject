## 2. 프롬프트 설계 — 5개 카테고리별 규칙 정리

- [x] **2-1 회사명 치환** — 최종 설계: "AI는 분류만, 찾기·치환은 코드가 결정적으로 처리"
  - **1차 설계(폐기)**: AI에게 "본문에서 회사명이 나오는 위치를 찾고, 문맥별로 치환 여부까지 판단해라"라고 맡겼더니, 같은 회사명이 여러 문장에 반복될 때 일부를 놓치는 버그 발생 (실제 테스트: 4곳 중 2곳만 잡힘). "찾기"와 "판단"을 동시에 시키는 게 원인 — 찾기는 AI가 완벽히 보장할 수 없는 작업.
  - **최종 설계**: 역할을 분리한다.
    1. AI는 본문에 언급된 회사명을 나열하고, 각각을 `worked_at`(경력 설명 문맥) / `target`(지원동기 등 그 외 문맥 — 치환 대상)으로만 분류한다. "몇 번 나오는지"는 신경 쓰지 않아도 됨 — 회사명당 한 번만 판단.
    2. 코드(`buildCompanySuggestions()`)가 `target`으로 분류된 회사명에 대해, 본문을 문장 단위로 쪼갠 뒤 그 문자열이 포함된 **모든 문장을 결정적으로(문자열 검색) 찾아** 각각 치환 제안을 만든다. 개수 제한 없음, 절대 누락되지 않음.
  - **전제(팀 논의로 확정)**: 지원하는 회사와 이미 근무한 회사의 이름은 절대 같을 수 없다. 본문에 예시·비교 목적으로 언급된 회사명은 없다고 가정한다 (즉 `worked_at`이 아니면 전부 `target`). 이 전제 덕분에 `target` 회사명은 문맥 재판단 없이 발견되는 즉시 전부 치환해도 안전함.
  - **실제 API 테스트로 검증**: 회사명이 5개 문장에 걸쳐 반복되고 경력 문맥 1곳이 섞인 샘플에서, 경력 문맥은 그대로 유지되고 나머지 5곳 전부 개별 제안(`company`, `company-1` ~ `company-4`)으로 정확히 생성됨을 확인
  - **남은 리스크**: AI가 본문의 회사명 자체를 회사명으로 인식하지 못하는 경우(드문 케이스)는 여전히 놓칠 수 있음 — 실용적으로 충분하다고 판단해 현재는 추가 조치 없음
- [ ] **2-2 인재상·가치 제안**
  - 본문에서 기존에 지원했던 회사의 인재상/가치 관련 문장을 탐지하는 기준 정의
  - 목표 회사의 mission/value는 AI 자체 지식으로 추론 (0번 결정사항) — "모르면 일반론으로 답하고 단정하지 마라" 제약 문구 포함, `confidence` 필드로 추정 여부 표시
- [ ] **2-3 직군·포지션 치환**

  - "선택 가능하도록" 이라는 요구사항 반영 — 대안이 여러 개일 수 있는지, 아니면 1건 제안 후 사용자가 적용/미적용만 선택하는 기존 UX(`sugg__toggle`)로 충분한지 확인

- [ ] 모든 카테고리 공통: 자소서 본문에 실제로 존재하지 않는 내용을 지어내지 않도록(할루시네이션 방지) 명시적 제약 문구 추가

## 9. 제안 겹침(overlap) 문제 — 해결됨

- [x] **문제**: 같은 문장이 여러 카테고리(company/keyword/position/grammar/tone)에서 동시에 제안될 수 있음. 기존 `applySuggestions()`가 제안을 순서대로 하나씩 `.replace(original, suggestion)` 하는 구조라서, 앞 순서 카테고리가 먼저 문장 전체를 바꿔버리면 뒤 순서 카테고리의 `original`이 더는 본문에 존재하지 않아 **사용자가 적용을 눌러도 조용히 무시**되는 상황이 발생함.
  - 실제 사례: "카카오의 백엔드 개발자로서 안정적인 서비스를 만들고 싶습니다." 문장이 `company`, `position`, `grammar` 세 카테고리에서 동시에 잡힘
- [x] **1차 방안(폐기)**: "한 문장은 한 카테고리만 갖는다" (우선순위 밀리면 통째로 버림) — 검토 중 실제 사용 사례("삼성전자에서 개발자로 성장하고 싶습니다"처럼 회사명+직군이 같은 문장에 같이 있는 경우)에서 필요한 수정이 통째로 사라지는 부작용을 발견해 폐기함.
- [x] **최종 해결**: 문제의 본질은 "같은 문장"이 아니라 "문장 전체 단위로 치환"하는 것이었음. `js/lib/matcher.js`에 `diffSpan()`(문장에서 실제로 달라지는 글자 구간만 계산)과 `computeSuggestionEdits()`(본문 안에서 그 구간의 절대 위치를 계산 + 진짜로 위치가 겹칠 때만 `CATEGORY_ORDER` 우선순위로 하나만 남김)를 추가. `applySuggestions()`는 이제 문장이 아니라 **정확한 글자 위치**를 뒤에서부터 치환하므로, 같은 문장 안에 있어도 위치가 안 겹치면(예: 회사명과 직군) 둘 다 정상 반영됨.
- [x] `js/features/result.js`의 미리보기 하이라이팅(`buildSegments`)도 동일한 `computeSuggestionEdits()`를 재사용하도록 변경 — 부수적으로 "회사명만 문장 전체로 하이라이트되던" 버그도 같이 해결됨 (하이라이트 구간과 실제 적용 결과가 항상 일치).
- [x] 실제 테스트로 검증: "삼성전자에서 개발자로 성장하고 싶습니다." (target: 토스, role: PM) → "카카오에서 PM으로 성장하고 싶습니다." 로 회사명·직군 둘 다 정상 반영됨을 확인.

## 10. 회사명 치환 추가 개선 (조사 교정 · position 확장 · 정확도)

- [x] **조사(은/는, 이/가, 을/를, 과/와, 로/으로) 자동 교정**: 회사명 치환 시 받침 유무에 따라 조사가 어색해지는 문제(예: "카카오와" → "토스와"는 맞지만 받침 있는 회사로 바뀌면 "삼성은" 처럼 교정 필요) 발견. `js/lib/josa.js`를 새로 만들어 한글 유니코드 받침 판별(`hasBatchim`, ㄹ받침 예외 포함)로 해결. 유닛 테스트로 은/는·이/가·을/를·과/와·로/으로 전부 검증 완료.
- [x] **제안 카드에 전체 문장 대신 바뀐 부분만 표시**: `diffSpan()`으로 원본/제안 문장의 공통 앞뒤를 잘라내고 실제로 달라진 구간만 카드에 노출하도록 `result.js` 수정.
- [x] **position(직군) 카테고리에도 경력 문맥 보존 규칙 적용**: company에만 있던 "화자 자신의 실제 경력은 건드리지 않는다" 원칙이 position엔 없어서, "삼성전자에서 개발자로 근무했습니다"의 "개발자"가 잘못 바뀔 위험이 있었음. company와 동일한 원칙 + 예시를 position 프롬프트에도 추가.
  - **주의(실제로 겪은 이슈)**: 처음 추가한 예시가 company 섹션의 예시와 회사명·직무명이 겹쳐서("삼성전자에서 개발자로 ~") 프롬프트 내 다른 섹션끼리 간섭을 일으켜 회사명 분류 정확도가 일시적으로 0%까지 떨어짐. 예시에 쓰는 회사명/직무명을 다른 것(네이버/마케터)으로 바꿔서 해결. (자세한 과정은 `AI_RELIABILITY_NOTES.md` 참고)
- [x] **회사명 분류 다수결(self-consistency) 도입**: 같은 문장도 AI가 매번 다르게 판단하는 비결정성 문제(측정: 동일 문장 10회 중 실패 빈도 큼)를 완화하기 위해, `companies` 분류만 담당하는 별도의 가벼운 프롬프트(`buildCompanyPrompt`)를 만들어 **3회 병렬 호출 후 다수결**로 최종 판단(`classifyCompanies()`). 동점이면 `confidence: low` 후보로 남김. 도입 후 동일 문장 10/10 정확도로 검증. 순차 호출이면 3배 느려지지만 `Promise.all` 병렬 호출로 응답 시간 증가를 최소화함 — 실측 스크립트(`scripts/compare-timing.mjs`, 발표용으로 캡처 후 삭제)로 검증.
- [x] **position `original`이 짧은 구절이라 잘못된 위치에 적용되는 버그 발견 및 수정**: position이 `original: "개발자로"`처럼 문장 전체가 아닌 짧은 구절만 반환하면, 본문에 같은 표현이 여러 번 있을 때 `text.indexOf()`가 항상 첫 번째(의도하지 않은) 위치를 찾아버림. 실제 사례: "삼성전자에서 개발자로 근무했습니다. 카카오에서 개발자로 근무하고 싶습니다."에서 뒷문장을 의도한 직군 제안이 앞문장(경력 문장)에 잘못 적용됨. company처럼 position/keyword/grammar/tone 모두 `original`을 반드시 "수정이 포함된 문장 전체"로 반환하도록 프롬프트 공통 규칙에 명시해 해결.
- [x] **카테고리별 suggestion이 서로의 담당 영역을 침범해 겹침 처리에서 탈락하는 버그 발견 및 수정**: position이 `suggestion`을 작성할 때 직무명뿐 아니라 회사명까지 같이 자연스럽게 바꿔 적으면서, `diffSpan()` 계산 범위가 회사명 부분까지 넓어져 company의 수정 범위와 겹침 → 우선순위에 밀려 position 수정이 통째로 버려짐. "각 카테고리는 자기 담당 부분만 수정하고 나머지는 원문 그대로 유지"하도록 프롬프트에 명시해 해결.
- [x] **position 분류에도 다수결(self-consistency) 확장**: company와 완전히 분리된 별도 프롬프트(`buildPositionPrompt`)로 3회 병렬 호출. company는 "회사명별 target/worked_at 득표"로 집계했지만 position은 매번 다른 문장을 고를 수 있어 방식이 다름 — "어떤 문장(original)을 제안했는지"로 묶어 가장 많이 나온 것을 채택하고, 2표 이상 동의해야 `confidence: high`(`classifyPosition()`). company·position·keyword/grammar/tone 세 그룹이 서로 독립적인 프롬프트/함수라 position에 다수결을 추가해도 company 로직·정확도에는 영향 없음. 5회 반복 테스트로 검증(경력 문맥 보존 + 회사명·직군 동시 반영 5/5).
- [ ] keyword/tone 카테고리에도 position과 같은 "경력 문맥 보존" 원칙이 필요한지 점검 (아직 미확인)
